General Introduction
In recent years, the world of software development and IT operations has changed
significantly. In the past, teams worked in silos, developers focused only on writing code,
while operations teams handled deployment and infrastructure. This separation created
many challenges: slow releases, communication gaps, frequent bugs, and complicated
manual processes that were hard to scale or maintain.
The rise of cloud computing and DevOps practices has transformed the way mod-
ern applications are built and deployed. With the cloud, companies can access flexible,
on-demand infrastructure and pay only for what they use. At the same time, DevOps
has brought development and operations teams closer together, promoting automation,
collaboration, and continuous delivery.
In this new approach, the goal is to deliver software faster and with better quality. De-
vOps encourages automation across the entire the software lifecycle starting from writing

DevOps Fundamentals and Tooling
Introduction
Today’s software teams are expected to deliver new features quickly, fix bugs rapidly,
and keep the system stable. DevOps is a culture and set of tools that aids in bringing the
two teams together - Development and Operations. It focuses on automation, continuous
integration and better collaboration.
DevOps uses automation, monitoring, infrastructure as code (IaC), containers, and
scalable deployment tools to solve challenges of slow releases, poor communication, and
manual work.
In this chapter we will examine some of the important concepts of DevOps such as
CI/CD, container orchestration, cloud computing, monitoring. We will compare some
tools and illustrate why certain of them is leveraged in this project by showing how they
fit in for a complete DevOps pipeline.
2.1
DevOps Overview
2.1.1
Definition of DevOps
DevOps is the combination of two words – Development Operations. It both a culture
and a set of practices which intends to improve software development, testing, delivery,
and maintenance. DevOps breaks down the silos between developers and operations and
encourages the two groups to work closely together. In other words, they are involved
together throughout the lifecycle of software development.
From writing the code to
running it, the two functions work hand in hand.
DevOps aims to increase software delivery speed while enhancing the stability, quality,
and security of applications. We can accomplish this through automation, CI/CD sys-
tems, live monitoring, and feedback loops. DevOps enables teams to react rapidly to user
requests, fix bugs more rapidly, and launch new features more frequently.
At its core, DevOps is based on a few important principles:
• Collaboration – Developers, testers, and system admins work as one team.
• Automation – Repetitive tasks like testing, building, and deployment are auto-
mated.
• Continuous Feedback – Teams use monitoring and logging to get feedback and
improve.

• Shared Responsibility – Everyone is responsible for both the development and
the performance of the application.
In short, DevOps helps create a fast, reliable, and efficient development process by
combining people, processes, and tools in a smart way [1].
2.1.2
DevOps Toolchain
- Plan: This is the first stage where teams develop projects after collecting requirements
and setting goals to determine construction methods.
- Code: The developers collaborate through version control tools with code manage-
ment options like Git to achieve smooth teamwork.
- Build: Code transformation into executable code begins after developers complete
their writing of program code.
- Test: The Test phase includes automated tests which verify the code operates as
intended and tries detecting any potential errors.
- Deploy: The software is released through a continuous deployment (CD) pipeline,
allowing frequent updates and new features to go to production servers, to the customer.
- Operate: The operations team leads software execution inside real environments to
verify its functionality for end users.
- Observe and Monitor: System monitoring tools enable teams to observe perfor-
mance while detecting issues so problems can be resolved efficiently [2].

Virtualization and Containerization
Virtualization
Single computers can execute multiple virtual machines due to the virtualization tech-
nology. Every virtual machine operates independently like a standalone system containing
its operating system and software collection.
The hypervisor functions as special software to develop manage virtual machines. A
physical computer operates multiple operating systems simultaneously thanks to this tech-
nology which ensures proper system isolation. Two examples of hypervisor software exist
in the market such as VMware and VirtualBox.
The practice of virtualization extracts maximum efficiency from one computer system
while reducing the need for multiple ones.

Containerization
Containerization permits an application to run in isolated environments aka containers
which is a light-weight form of virtualization. Virtual machines use hypervisor to virtu-
alize guest OS. Hence, they are heavier than containers. Container only packages and
virtualizes app library. Containers bundle the application code and its dependencies so
they will work uniformly no matter where the container is deployed. With this, it can
provide key benefits such as faster startup times, better resource utilization and isolation.

GitOps and Deployment Automation
We will implement the GitOps method to be part of our Continuous Deployment (CD)
strategy. It’s the practice of using Git as the source of truth for everything. GitOps en-
ables the management and automation of application delivery using Git-based workflows,
making deployment traceable, auditable and repeatable. Anyone who wants to make a
change to the system must first commit to Git; it will then automatically and securely
sync to the running environment.
The fundamental principles of GitOps are declarative, automated, versioned; whether
deployment or infrastructure. This technique makes it easy for teams to work together,
roll back codes, and make environments consistent by using an app that watches the
Git database to see what changes were made, which it will implement in the Kubernetes
cluster [5]. GitOps workflows support many tools the most famous being ArgoCD and
Flux. Although Flux is simple and flexible due to CLI-based management, ArgoCD is
more user-friendly and supports multi-application environments out-of-the-box and has a
web UI.
Selected Tool: ArgoCD
For this project, we chose ArgoCD to execute Continuous Deployment with GitOps.
ArgoCD is an open-source, Kubernetes-native tool that can continuously watch Git repos-
itories and apply the desired state automatically. It is a great tool for managing and
visualising Kubernetes deployments using GitOps. The ease of use, visual interface, and
strong ecosystem support which comes with this tool are the reasons why it was selected.

occurs.
The combination of observability and monitoring enables teams to get better control
over the systems.

request latency, which help in tracking system performance over time.
• Alerts: Notify teams in real time when specific thresholds or anomalies are detected,
enabling quick responses to potential incidents.

Docker is a free platform that lets you package an application into a container. This

running on the developer’s laptop or the production server. Docker is designed to solve
the "It works on my machine" problem. Thus, it manages to make the developers and
operations teams work in a consistent manner with significantly less friction
Use Cases of Docker
Docker provides significant advantages in various stages of the software development
lifecycle:
• Development and Testing: Developers can create isolated and consistent envi-
ronments on any system. This ensures uniformity across team members’ setups and
reduces environment-related bugs.
• Application Deployment: Docker containers encapsulate the application and its
dependencies, making deployment fast, reproducible, and environment-agnostic.

allowing for independent development, scaling, and maintenance of different parts of
the application.
• Multi-Cloud and Hybrid Cloud Environments: Docker increases application
portability, making it easier to move between different cloud providers like AWS,
Azure, or on-premise infrastructure.
• CI/CD Integration: Docker integrates seamlessly with CI/CD pipelines, enabling
efficient automation of builds, testing, and deployments with high repeatability.
Advantages of Docker
Docker introduces several benefits that make it a preferred tool in DevOps workflows:
• Portability: Containers can run on any system with Docker installed, regardless of
the underlying infrastructure or operating system.
• Rapid Deployment: Containers are lightweight and start almost instantly, en-
abling faster testing, development, and scaling of applications.
• Resource Efficiency: Docker containers consume fewer resources compared to tra-
ditional virtual machines, allowing better utilization of system hardware.

Kubernetes Fundamentals
Kubernetes provides a declarative approach to configuration using YAML files, enabling
developers to specify how their applications should run. Its core functionalities include:
• Desired state management:
You use YAML files to instruct Kubernetes regard-
ing the number of containers (Pods) you want it running. Along with their image,
how should it connect, and more. Kubernetes ensures this desired state is always
met.
• Automation: Kubernetes automates tasks such as scheduling, health monitoring,
and restarts.
• Self-healing: Kubernetes takes care of things when the enterprise software goes
down or if the whole is down. It will replace anything which has gone down.
• Scalability: You can easily update the number of replicas in a Deployment file to
scale your applications further up or down.
• Portability: Kubernetes can be used in multiple environments including your laptop
(Minikube), public clouds (AWS, GCP, Azure) and private data centres [9].
Because of these, Kubernetes is ideal to run a production environment, especially with
microservices or large applications.
Key Concepts
Some of the fundamental components in Kubernetes include:
• Pod: A pod is the smallest unit in k8s. It usually runs one container, but can run
more if needed. Containers in the same pod all share the same IP and storage space.
Kubernetes actually runs Pods when you deploy an app there.
• Deployment: A Deployment is that controller which manages Pods. It guarantees
that a certain count of Pods are continuously operating. If one crashes, the Deploy-
ment creates a new one. Through deployments, you can apply rollouts and rollbacks
as well.
• ReplicaSet: A ReplicaSet makes sure a certain number of Pods are always run-
ning. While Deployments manage ReplicaSets, you can also use ReplicaSet singu-
larly, though this is not common. Deployments are preferred for higher-level control.
• Services: Services help because Pods come and go, changing their IPs. A Service
provides a stable way to access a set of Pods. There are different types; ClusterIP
Internal access only. NodePort exposes the service on each node’s port. LoadBal-
ancer opens the service to the web utilizing a load balancer from the cloud provider.
• Ingress: Ingress manages external access to the services in a Kubernetes cluster,
typically HTTP/HTTPS traffic. It provides routing rules and allows you to expose
multiple services under a single IP, offering more control over traffic routing.
• Namespaces: Namespaces help you organize objects within your cluster. For in-
stance, having a separate "dev" namespace and "prod" namespace isolates the envi-
ronments.
• ConfigMaps and Secrets: ConfigMaps help in saving non-sensitive data the ap-
plication use, like URLS, file paths, etc. Secrets are like ConfigMaps but for storing
private data like passwords, access tokens, and ssh keys [9].

Kubernetes Cluster Architecture
Kubernetes runs in a cluster made up of multiple machines (nodes). It follows a master-
worker architecture, where some nodes manage the cluster (control plane), and others run
the applications (worker nodes).
Control Plane (Master Node)
: The control plane is the brain of the Kubernetes
cluster. The control plane manages everything and makes critical decisions for cluster-
wide behavior.
• kube-apiserver: is the interface for the Kubernetes control plane. This API server
processes all commands and requests coming from users, CLI tools, or dashboards.
• etcd: It is a distributed key-value store that stores the cluster state and configura-

• kube-scheduler:is responsible for choosing a worker node to run a pod.
• controller-manager: Runs different controllers that handle tasks like checking if a

Worker Nodes:
Responsible for running containers.

• kube-proxy: The program used to manage the network rules for Kubernetes Ser-

This architecture allows Kubernetes to be scalable, resilient, and highly available.

ArgoCD
ArgoCD is a Continuous Delivery tool for Kubernetes, and it is declarative and GitOps-
based. It is part of Argo Project, a CNCF (Cloud Native Computing Foundation) project.
ArgoCD is a tool that automates the management and synchronization of Kubernetes
resources based on Git changes.
Instead of using a push-based deployment strategy, ArgoCD uses a pull-based approach
where it pulls the application configuration from a Git repository and applies it in the
target Kubernetes cluster. With the help of this GitOps model, infrastructure and appli-
cation tracing, stability and simple connectivity are improved.

Main Features of ArgoCD
• Git as the Source of Truth: With ArgoCD, the Git repository has all Kubernetes
configuration (manifests). So, your Git repo is the "single source of truth" for your
deployments. Argo CD keeps an eye to your Git repo for changes, making sure that
the deployed state of your application and the Git state is in sync. Fewer mistakes
and greater uniformity across setups.
• Declarative Application Management: DevOps engineers describe the desired
state of their applications with YAML manifests or Helm/Kustomize templates. Ar-
goCD always compares this desired state to the actual state in the cluster and alerts
or acts when there is a mismatch. This makes deployments predictable and repro-
ducible.
• Web UI, CLI, and API Access: ArgoCD comes with a web UI interface that
allows for easy access and management of clusters. visualize deployment. Besides,
you can access the REST and command-line interface argocd. API for scripting and
automation. The user interface shows the sync status, differ-. You have the option
to rollback if there are differences.
• Automatic Sync and Health Checks: ArgoCD can sync the application without
requiring user intervention. Automatically, when it observes a change in Git. It also
checks the health of Kubernetes. Resources to make sure it is running fine. This
allows teams to automate the full. deployment cycle [10].
Advantages in CI/CD Automation
Using ArgoCD in CI/CD pipelines enhances the delivery process by introducing GitOps
principles in the CD phase:
• Separation of CI and CD: CI tools such as jenkins or GitHub Actions focus on
building, testing, and pushing container images. ArgoCD manages the deployment by
monitoring Git repositories and synchronizing updates with the Kubernetes cluster.
It helps with maintanability and security issues.
• Improved Security: As ArgoCD is pull-based, it doesn’t require external CI/CD
tools to have direct access to the Kubernetes cluster. ArgoCD fetches modifications
from Git and implements them within the cluster, minimizing attack surfaces.
• Safe and Predictable Deployments: Every deployment is tracked through Git.
Rollbacks are as simple as reverting a commit. ArgoCD checks that the live system
matches the state defined in Git.
• Real-Time Monitoring: The UI lets DevOps engineers and developers know ex-
actly what is deployed, and how it has changed. It shows variations between the Git
and cluster state and gives the option to deny or approve syncs.
• Scalability and Team Collaboration: ArgoCD can be used effectively by many
teams trading in multiple applications. Every team manages its apps and Git repos
without affecting the others [10].

Prometheus
Prometheus is an open-source monitoring and alerting system that is designed for cloud-
native environments. It is designed to collect, process and re store real-time metrics from
different systems and applications. Prometheus is widely used in DevOps workflows. It
helps ensure services are available and performant by enabling teams to observe their
systems behaviour over time.
Prometheus is composed of several key components that work together to offer complete
monitoring capabilities:
• Prometheus Server: The core component that scrapes metrics from configured
targets at specified intervals and stores them locally.
• Time-Series Database (TSDB): A highly efficient storage engine used to store
collected metrics in a format optimized for time-series queries.
• PromQL: A query language used to extract, filter, and aggregate data from the
TSDB.
• Alertmanager: Responsible for handling alerts generated by Prometheus rules. It
supports deduplication, grouping, and routing to various notification channels like
email, Slack, or PagerDuty.
• Exporters: Lightweight components or agents that expose metrics from third-party
systems like databases, message queues, or web servers in a format Prometheus
understands.
• Pushgateway: Allows ephemeral or batch jobs to push metrics to Prometheus when
scraping is not feasible.
• Web UI: A built-in graphical interface used to run PromQL queries, view targets
and alerts, and explore metrics data [13].

AWS Architecture

We built our system inside a Virtual Private Cloud (VPC), which gave us a safe and
private network inside AWS. This helped us control how different parts of our system
talked to each other and kept everything secure.
To ensure high availability and fault tolerance, we deployed the infrastructure across
two separate Availability Zones (AZ1 and AZ2). These zones are distinct physical data
centers within the same region. By spreading our components across multiple AZs, we
made the system resilient against failures in any single zone, which improved the reliability
of our application.
In each availability zone, we created two types of subnets: public and private. The
public subnets were for components that needed to connect to the internet.
We put
Jenkins here to handle automated tasks like building code and running tests. We also
placed SonarQube in the public subnet to check our code quality and find errors early.
The Nexus repository was also here, which stored our artifacts.
To securely access resources in the private subnets, we used a Bastion host as a secure
gateway. This host allowed authorized users to connect via SSH to internal servers without
exposing those servers directly to the internet.
The private subnets contained critical infrastructure components that were not pub-

licly accessible.
We ran our Kubernetes cluster, created with kubeadm, within these
private subnets to increase security by limiting direct internet exposure. The cluster was
supported by an Auto Scaling Group (ASG), which automatically adjusted the number
of worker nodes based on workload demands. This autoscaling ensured that our cluster
maintained optimal performance and cost-efficiency by dynamically matching resources
to current needs.
Because the private subnets had no direct internet access for security reasons, we

outbound internet connections for tasks like downloading updates or pulling container

To handle user traffic, we set up an Application Load Balancer (ALB). The ALB

to the different applications running in the cluster. This setup helped our app handle

and secure.
In short, we designed our system to be secure, flexible, and easy to manage. Public
tools like Jenkins and SonarQube were accessible but safe, while our main app ran in a
private, protected network. We used autoscaling and load balancing to handle changing
workloads, and NAT Gateways to allow secure internet access for our private servers.

• Auto Scaling Group
After the launch template was created, we created the ASG to manage the Kubernetes
worker nodes. This makes our cluster highly scalable and fault-tolerant. The auto-scaling
group has been configured to have minimum instances set to 2, maximum instances set to
4 and desired capacity set to 2. This ensures that there are always EC2 instances running
as workers. These instances in two private subnets in different availability zones for fault
tolerance and high availability.
The ASG will use the launch template we created above.
That includes the AMI
(baked with Kubernetes components using Packer), instance profile (IAM permissions),
security groups, EBS configuration, and most importantly a kubeadm join command in
the user-data script. Each instance created by the ASG joins the Kubernetes cluster (as
a worker node) automatically on launch.
In the Auto Scaling Group we add certain tags to enable Cluster Autoscaler support:
• k8s.io/cluster-autoscaler/enabled = true : tells the Cluster Autoscaler that
this ASG should be managed.
• k8s.io/cluster-autoscaler/kubernetes = owned : tells CA this ASG belongs to
the current k8s cluster named kubernetes.
Workflow and Trigger Mechanism
When the Kubernetes cluster doesn’t have enough resources to run new pods, the CA
running inside the cluster notices this. It checks the ASG that is tagged for autoscaling.
If more nodes are needed, the CA uses its IAM permissions to ask the ASG to create
a new EC2 instance.
The ASG launches a new instance using the Launch Template we defined. This instance
runs a startup script with the kubeadm join command, which makes it automatically join
the cluster as a worker node.
This process allows the cluster to scale automatically, handle more workloads when
needed, and save costs when fewer nodes are required without any manual steps.

build a scalable, automated, and efficient solution that supports continuous delivery from
code to production.
